# 数据库与数据模型

本文根据 `prisma/schema.prisma` 总结当前代码版本的数据模型。这里只解释模型用途，不包含任何真实数据。

## 1. 用户与认证

| 模型 | 用途 |
| --- | --- |
| `User` | 系统用户和选手基础信息，包括邮箱、昵称、头像、积分、ELO、胜负场、角色、封禁状态和邮箱验证状态。 |
| `EmailVerificationToken` | 邮箱验证和密码重置 token 的哈希记录，包含过期时间和消费时间。 |
| `AdminTrustedDevice` | 管理员二次验证中的信任设备记录，保存 token hash、过期时间和撤销状态。 |
| `UserIdentity` | 参赛证明所需姓名和学号的哈希记录。代码注释说明该信息用于生成证明，不以明文反查。 |

维护注意事项：

- 不要直接在数据库中写入明文密码。
- 不要把真实姓名、学号、邮箱列表导出到公开文档中。
- 封禁用户、管理员角色、邮箱验证状态都属于敏感管理字段。

## 2. 比赛与报名

| 模型 | 用途 |
| --- | --- |
| `Match` | 比赛或快速比赛主体，包含标题、描述、时间、地点、类型、状态、赛制、报名截止时间、创建人等。 |
| `Registration` | 报名记录，关联比赛和用户，记录报名角色、报名状态和创建时间。 |
| `MatchGrouping` | 比赛分组结果，使用 JSON payload 保存分组、晋级配置和淘汰赛签表等结构。 |

相关枚举：

- `MatchType`：`single`、`double`、`team`。
- `MatchStatus`：`registration`、`ongoing`、`finished`。
- `CompetitionFormat`：`group_only`、`group_then_knockout`。
- `RegistrationRole`：`player`、`captain`、`substitute`。
- `RegistrationStatus`：`registered`、`cancelled`、`confirmed`。

维护注意事项：

- `Match.isQuickMatch` 用于区分正式比赛和快速比赛。
- `MatchGrouping.payload` 是 JSON，结构变更前应兼容旧数据。
- 删除比赛会级联删除报名、赛果、分组等关联数据，操作前应备份。

## 3. 双打组队

| 模型 | 用途 |
| --- | --- |
| `MatchDoublesTeam` | 双打小队主体，关联比赛和创建人。 |
| `MatchDoublesTeamMember` | 双打小队成员，记录队伍、比赛、用户和成员槽位。 |
| `MatchDoublesInvite` | 双打邀请记录，包含邀请人、被邀请人、比赛和邀请状态。 |

维护注意事项：

- 双打比赛报名依赖组队流程。
- 邀请状态目前是字符串字段，后续如扩展状态建议统一枚举或常量。

## 4. 赛果、ELO 与积分

| 模型 | 用途 |
| --- | --- |
| `MatchResult` | 比赛结果，记录胜方、负方、队伍成员、比分 JSON、上报人、确认状态和确认人。 |
| `EloHistory` | ELO 变动历史，记录变动前后分数、delta、关联比赛和赛果。 |
| `PointsTransaction` | 积分流水，记录积分增减、余额、类型、原因和 referenceId。 |

相关枚举：

- `PointsTransactionType`：`earn`、`spend`、`adjustment`、`refund`。

维护注意事项：

- ELO 和积分是可追溯数据，不建议直接手工改用户总分。
- 如需纠错，应优先通过系统中的纠错逻辑或记录清楚修改原因。
- `referenceId` 用于避免重复发放某些比赛积分，变更规则时要小心。

## 5. 排行、荣誉与评价

| 模型 | 用途 |
| --- | --- |
| `Badge` | 徽章定义。 |
| `UserBadge` | 用户获得徽章记录。 |
| `LeaderboardCache` | 排行榜缓存记录，包含 scope、rank、用户数据和快照时间。 |
| `RankingSnapshot` | 排名快照 JSON。 |
| `Review` | 用户对用户的评价，关联比赛，包含 scores 和 comment。 |

维护注意事项：

- 当前代码中能看到排行榜页面读取用户实时数据和徽章数量。
- `LeaderboardCache`、`RankingSnapshot`、`Review`、奖励兑换相关模型在 schema 中存在，但当前路由中没有看到完整前台页面。线上是否使用需维护者确认。

## 6. 奖励与兑换

| 模型 | 用途 |
| --- | --- |
| `Reward` | 奖励定义，包含标题、描述、所需积分、库存、类型和元数据。 |
| `RewardRedemption` | 用户兑换记录，记录奖励、用户、消耗积分、状态和创建时间。 |

相关枚举：

- `RewardType`：`physical`、`digital`、`coupon`。

维护注意事项：

- 当前代码版本中模型存在，但没有看到完整兑换页面。若未来启用，需要补充库存、审核、发放和撤销规则。

## 7. 审计与站点状态

| 模型 | 用途 |
| --- | --- |
| `AuditLog` | 管理员操作日志，记录操作者、动作、对象、IP、User-Agent、details 和时间。 |
| `SiteSetting` | 站点开关，当前包含 `isClosed`。 |

维护注意事项：

- 审计日志是排查误操作的重要依据，不建议随意清空。
- 站点关闭功能适合维护或紧急暂停普通用户访问。

## 8. 参赛证明

| 模型 | 用途 |
| --- | --- |
| `ParticipationCertificate` | 参赛证明记录，关联比赛和用户，保存唯一证明编号。 |

维护注意事项：

- 证明编号应保持唯一。
- 证明导出依赖用户身份哈希、比赛参与资格和 PDF 字体配置。

## 9. 迁移注意事项

- 开发环境使用 `npm run prisma:migrate`。
- 生产环境应使用 `npx prisma migrate deploy`。
- 修改 schema 前应先备份数据库。
- JSON 字段结构变更需要考虑旧数据兼容。
- 涉及 ELO、积分、赛果的迁移应先在测试数据库验证。
