# 权限与安全

本文总结当前代码版本中的权限和安全设计，并列出维护时必须注意的事项。

## 1. 账号体系

当前代码使用邮箱和密码登录。注册后需要通过邮件完成邮箱验证，未验证用户不能作为当前登录用户使用。

密码使用 `scrypt` 哈希，不应也不需要保存明文密码。

## 2. Session

登录成功后，系统会写入 `ustc_tta_session` cookie。该 cookie：

- 使用 HMAC 签名防篡改。
- 设置 `httpOnly`，前端 JS 不能直接读取。
- 生产环境设置 `secure`。
- 设置 `sameSite: lax`。

维护注意事项：

- `AUTH_SECRET` 必须足够随机，不能泄露。
- 更换 `AUTH_SECRET` 会让所有旧登录态失效。
- 如果管理员重置用户密码或封禁用户，应确认旧登录态是否按当前代码预期失效。

## 3. 管理员权限

用户角色由 `User.role` 控制，取值包括普通用户和管理员。管理员后台入口为 `/admin`。

管理员后台包含：

- 邮箱二次验证。
- 信任设备。
- 用户封禁、删除、资料编辑、角色修改。
- 批量创建测试账号。
- 批量加入比赛。
- 网站开关。
- 审计日志查看。

维护注意事项：

- 管理员账号不要共享。
- 离任管理员应及时降权或封禁。
- 管理员邮箱必须可用，否则无法完成二次验证。
- `ADMIN_REAUTH_SECRET` 建议单独配置；如未配置，代码会回退使用 `AUTH_SECRET`。

## 4. CSRF 防护

项目包含 CSRF token 机制：

- `/api/csrf-token` 设置并返回 token。
- `CsrfFormInjector` 会向表单注入隐藏字段。
- Server Actions 和部分 API 会调用 `validateCsrfToken`。

维护注意事项：

- 新增写操作时应调用 `validateCsrfToken`。
- 不要为了省事跳过 CSRF 校验。
- 如果表单提交出现安全校验失败，优先检查 token 注入、cookie 和部署域名。

## 5. 中间件访问控制

`middleware.ts` 会：

- 跳过静态资源和 Next.js 内部资源。
- 放行登录、CSRF token、站点状态、关闭页和管理员页。
- 检查站点是否关闭。
- 对未登录用户跳转到 `/auth`。

注意：中间件只做入口层控制，真正的权限判断仍应放在 Server Actions 和服务端数据读取中。

## 6. Server Actions 权限

多数写操作会在服务端重新获取当前用户，并判断：

- 是否已登录。
- 是否为比赛创建者。
- 是否为管理员。
- 是否为对阵双方。
- 是否处于允许操作的比赛阶段。

维护时不要只在前端隐藏按钮。前端按钮隐藏只能改善体验，不能作为安全边界。

## 7. 敏感环境变量

以下变量属于敏感或半敏感配置：

- `DATABASE_URL`
- `DATABASE_URL_UNPOOLED`
- `AUTH_SECRET`
- `ADMIN_REAUTH_SECRET`
- `AZURE_COMMUNICATION_CONNECTION_STRING`
- `CLOUDINARY_API_KEY`
- `CLOUDINARY_API_SECRET`

要求：

- 不要提交 `.env`。
- 不要把真实值写入 README、docs、Issue、PR 或聊天记录。
- private repo 也不应保存真实密钥。
- 发现泄露后应立即轮换密钥和数据库密码。

## 8. 图片上传安全

头像上传当前包含：

- 文件大小限制。
- MIME 类型判断。
- 图片 magic bytes 校验。
- 上传到 Cloudinary。

维护注意事项：

- 不要放宽上传类型到任意文件。
- 不要直接信任用户提交的 URL。
- Cloudinary 密钥只能放在服务端环境变量中。

## 9. PDF 证明与身份信息

参赛证明功能会要求用户输入姓名和学号。代码中以哈希形式保存身份信息，用于之后验证输入是否一致。

维护注意事项：

- 不要在日志、文档或前端页面展示完整身份信息。
- 证明导出接口应保持登录和资格校验。
- 字体路径错误可能导致导出失败，但不应把服务器路径暴露给普通用户。

## 10. 审计日志

管理员关键操作会写入 `AuditLog`，包括动作、对象、操作者、IP、User-Agent 和 details。

维护注意事项：

- 不建议随意清理审计日志。
- 如需清理，应先导出备份并记录原因。
- 审计日志中可能含用户邮箱或对象标识，不应公开展示。

## 11. 当前安全限制与建议

当前代码已经有基础防护，但仍建议持续改进：

- 登录限流是进程内 Map，多实例部署时不共享，建议改为 Redis 或数据库限流。
- CSP 仍包含 `unsafe-inline` 和 `unsafe-eval`，后续可评估更严格策略。
- 管理后台高风险操作建议增加二次确认和更细粒度权限。
- 部署平台、数据库和第三方服务账号应启用强密码和多因素认证。

## 12. 新增功能时的安全检查清单

- 是否需要登录？
- 是否需要管理员或比赛创建者权限？
- 是否调用了 CSRF 校验？
- 是否会修改 ELO、积分、赛果或用户角色？
- 是否会处理上传文件？
- 是否会输出用户邮箱、姓名、学号等信息？
- 是否需要写入审计日志？
- 是否需要限制频率？
