# 技术架构

本文总结当前代码版本能确认的技术组成。未能从代码直接确认的部分会标注“需维护者确认”。

## 1. 总体架构

项目是一个 Next.js 全栈应用：

- 前端页面由 React 组件和 Next.js App Router 组织。
- 写操作主要通过 Server Actions 完成。
- API routes 用于 CSRF token、站点状态和 PDF 证书导出等场景。
- 数据库访问通过 Prisma Client。
- 数据库存储使用 PostgreSQL，配置中推荐 Neon 或其他 PostgreSQL 服务。

## 2. 前端

| 项目 | 当前使用 |
| --- | --- |
| 框架 | Next.js 16 App Router |
| UI | React 19 |
| 样式 | Tailwind CSS 4 |
| 图标 | lucide-react |
| 图表 | Recharts |
| 日期选择 | react-day-picker |
| 组件组织 | `src/components` 按业务域拆分 |

前端主要是深色竞技风格界面，包含桌面侧栏和移动端顶部导航。首页、比赛大厅、排行榜、比赛详情和管理后台都使用响应式布局。

## 3. 后端与业务逻辑

Next.js Server Actions 承担主要后端写操作，包括：

- 注册、登录、邮箱验证、密码重置。
- 创建比赛、报名、退赛、编辑比赛。
- 分组预览、确认分组、赛果登记、赛果确认、赛果纠错。
- 快速比赛登记、确认、拒绝和超时清理。
- 双打邀请发送、接受、撤回。
- 管理员用户管理、角色管理、站点开关、测试账号创建和批量操作。

少量 API routes 用于：

- `/api/csrf-token`：生成或返回 CSRF token。
- `/api/site-status`：返回站点是否关闭。
- `/api/matchs/[id]/certificate`：生成参赛证明 PDF。

## 4. 数据库

| 项目 | 当前使用 |
| --- | --- |
| ORM | Prisma |
| 数据库 | PostgreSQL |
| 迁移目录 | `prisma/migrations` |
| Prisma schema | `prisma/schema.prisma` |

主要数据包括用户、比赛、报名、赛果、ELO 历史、积分流水、双打队伍、组队邀请、参赛证明、审计日志等。

## 5. 认证与权限

认证采用邮箱和密码登录：

- 密码使用 Node.js crypto 的 scrypt 派生哈希。
- 登录后写入签名 session cookie。
- 邮箱验证和密码重置通过邮件 token 完成，数据库中保存 token hash。
- 管理员后台要求管理员角色，并包含邮箱二次验证和信任设备机制。
- 写操作普遍使用 CSRF 校验。

维护者应重点保护 `AUTH_SECRET`、`ADMIN_REAUTH_SECRET`、数据库连接和邮件服务密钥。

## 6. 文件、图片与 PDF

| 类型 | 当前实现 |
| --- | --- |
| 头像上传 | Cloudinary |
| 图片校验 | 文件大小、MIME、图片 magic bytes |
| 参赛证明 | PDFKit |
| PDF 字体 | `CERTIFICATE_FONT_PATH` 指向字体文件 |

参赛证明接口需要字体文件可被部署环境读取。`next.config.ts` 中已为证书接口包含字体和 PDFKit 数据文件。

## 7. 邮件服务

代码使用 Azure Communication Email：

- 注册后发送邮箱验证邮件。
- 账号帮助中发送验证或重置密码邮件。
- 管理员后台发送二次验证验证码。

是否使用其他邮件服务或备用服务，当前代码未看到完整实现，需维护者确认。

## 8. 部署相关

当前仓库包含两类部署线索：

- `netlify.toml`：Netlify 构建命令为 `npm run prisma:generate && npm run build`，并使用 `@netlify/plugin-nextjs`。
- README 原有内容和常见运维流程中提到自建服务器、PM2、反向代理等方式。

如果使用自建服务器，通常流程是安装依赖、执行数据库迁移、构建、用 PM2 启动 `npm run start`，再由 Nginx 或其他反向代理提供 HTTPS。

## 9. 安全响应头

`next.config.ts` 设置了：

- Content-Security-Policy
- X-Content-Type-Options
- X-Frame-Options
- Referrer-Policy

CSP 仍包含 `unsafe-inline` 和 `unsafe-eval`，这可能是为了兼容当前构建和运行方式。若后续安全要求提高，应逐步评估是否能改为 nonce 或更严格策略。

## 10. 需要维护者确认的事项

- 线上部署平台是 Netlify、自建服务器、Vercel 还是其他平台。
- 线上数据库是否使用 Neon，以及是否使用连接池和 unpooled 连接。
- 证书 PDF 字体路径在生产环境是否正确配置。
- 奖励、兑换、评价、排行榜缓存等模型是否有线上业务使用。
- 当前仓库版本是否已经完全同步到线上。
